```
User Chat
    ↓
Rust App (RAG orchestrator)

    ├── GitHub repo index (chunked, embedded locally)
    ├── Local Ollama (Qwen2.5-Coder 7B) ← scaffold/stub gen
```

## The `.rustassistant/` Per-Repo Cache Concept

Each repo you register gets its own `.rustassistant/` directory that acts as a local knowledge layer:

```
.rustassistant/
├── manifest.json        # repo identity, last_synced, branch, remote URL
├── tree.txt             # rolling project tree snapshot (regenerated on sync)
├── symbols.json         # parsed functions, structs, traits, impls
├── todos.json           # extracted TODO/STUB/FIXME tags with file:line refs
├── embeddings.bin       # cached chunk embeddings (skip re-embedding on restart)
└── context.md           # human-readable summary of what this repo is/does
```

`tree.txt` is the killer feature — on every sync you run `find . -not -path '*/target/*' -not -path '*/.git/*'` and cache it. The chat interface always prepends the current tree to every code gen prompt so the LLM knows exactly where things live.

## Repo Registry Architecture

Add a `repos` table to your SQLite/Postgres schema:

```sql
CREATE TABLE registered_repos (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL,
    local_path  TEXT,           -- if cloned locally
    remote_url  TEXT,           -- github url
    last_synced DATETIME,
    branch      TEXT DEFAULT 'main',
    active      BOOLEAN DEFAULT true
);
```

Then a `RepoSyncService` that on startup (and on a cron/webhook trigger):
1. Walks the repo tree → writes `tree.txt`
2. Extracts all `// TODO:`, `// STUB:`, `// FIXME:` → writes `todos.json`
3. Parses `Cargo.toml` for crate name, deps, feature flags → into `manifest.json`
4. Chunks changed files → re-embeds → updates Redis

## What's Missing in the Repo Right Now

Based on the README + structure, the gaps for this specific use case are:
